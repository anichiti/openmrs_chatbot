## Understanding the Workflow Traces

### What is a Workflow Trace?

When you ask a question, the chatbot:
1. **Classifies** your question's intent (medication, allergy check, vitals, etc.)
2. **Fetches** patient data from OpenMRS database
3. **Routes** to the appropriate handler pipeline
4. **Processes** using specialized agents and AI models
5. **Generates** a personalized response

The **Workflow Trace** shows you EXACTLY what happened at each step.

### Reading the Workflow Trace Panel

After you receive a response, a **Backend Workflow Trace** panel appears below the message.

Click the panel header to expand it and see:

#### 1. Metadata Section (Top)
- **Trace ID**: Unique identifier for this query (useful for debugging)
- **Duration**: Total time to process the query (in milliseconds)
- **Intent**: What type of question was detected

Example:
```
Trace ID: abc123def456
Duration: 250ms
Intent: MEDICATION_QUERY
```

#### 2. Intent Classification Stage (🔍)
Shows how the question was analyzed:

**Example Flow:**
```
Intent Classification (45ms)
  └── TriageAgent
      └─ classify_intent: Detected ALLERGY_QUERY
      └─ Duration: 12ms
      
Substeps:
  → parse_question
    • intent: ALLERGY_QUERY
    • patient_id: 12345
    • user_role: DOCTOR
```

#### 3. Patient Data Fetch Stage (📊)
Shows data retrieved from the database:

## Different Question Types & Their Workflows

### 1. Direct Data Queries

**Question:** "What is the patient's age?"

**Workflow:**
- Intent: VITALS_QUERY
- Duration: ~50ms (FAST - no LLM needed)
- Sources: Patient Record (cached)

Direct queries skip the LLM processing entirely - they fetch answers directly from the database.

### 2. Medication Dosage Queries

**Question:** "What's the correct paracetamol dose for this 8-year-old?"

**Workflow:**
```
1. Intent Classification (12ms)
   → Detects MEDICATION_QUERY with dose keywords
   
2. Patient Data (150ms)
   → Fetches patient weight, age
   
3. Processing (280ms)
   → DoseCalculator: Calculates based on weight
   → RxNormAPISkill: Validates against FDA guidelines
   → ValidationAgent: Confirms safe dose range
   
4. Response: "400-600mg every 4-6 hours (max 3000mg/day)"
```

### 3. Allergy Checks

**Question:** "Is this patient allergic to penicillin?"

**Workflow:**
```
1. Intent Classification (15ms)
   → Detects ALLERGY_QUERY
   
2. Patient Data (180ms)
   → Fetches allergy records, medication history
   
3. Processing (120ms)
   → AllergyFetcher: Checks OpenMRS allergy records
   → CrossReactivityChecker: Checks drug families
   
4. Safety Net: 
   → Appended to ANY medication query
   → Warns if contraindicated
```

### 4. Immunization Records

**Question:** "What vaccines has this child received?"

**Workflow:**
```
1. Intent Classification (14ms)
   → Detects IMMUNIZATION_QUERY
   
2. Patient Data (200ms)
   → Fetches immunization history
   
3. Processing (95ms)
   → ImmunizationFetcher: Retrieves vaccine records
   → MilestoneValidator: Checks if age-appropriate
   
4. Response: Lists all vaccines with dates
```

### 5. Knowledge-Based Queries

**Question:** "What are the side effects of ibuprofen?"

**Workflow:**
```
1. Intent Classification (18ms)
   → Detects GENERAL_MEDICAL_QUERY
   
2. Patient Data (160ms)
   → Loads patient context (optional)
   
3. Processing (2500ms)
   → KnowledgeAgent: Searches medical KB
   → Ollama LLM: Generates detailed answer
   → ResponseAgent: Formats for doctor/patient
   
4. Response: Complete side effect list with severity
```

## Performance Insights

### Typical Response Times

| Query Type | Duration | Bottleneck |
|-----------|----------|-----------|
| Direct data (age, vitals) | 40-80ms | Database warmup |
| Allergy check | 200-350ms | OpenMRS query + validation |
| Medication dosage | 300-500ms | RxNorm API lookup + calculation |
| Immunization history | 250-400ms | Database query + milestone check |
| Knowledge queries | 2000-5000ms | Ollama LLM generation |
| Hybrid queries | 300-800ms | Combines multiple sources |

### Optimizing Performance

The workflow trace helps you identify slow components:

- **Slow Patient Data Fetch (>500ms)?** → Database is slow or needs indexing
- **Slow API calls (>1000ms)?** → Check network/API rate limits  
- **Slow LLM response (>3000ms)?** → Normal for Ollama, consider faster model
- **Multiple agent calls?** → Unnecessary agents might be redundant

## API Endpoints for Programmatic Access

### Get a Specific Trace
```
GET /api/trace/{trace_id}
Response: { trace: { metadata, stages, steps } }
```

### List All Recent Traces
```
GET /api/traces
Response: { traces: [ { trace_id, metadata }, ... ], total: N }
```

### Clear All Traces
```
POST /api/traces/clear
Response: { message: "All traces cleared" }
```

### Core Chat Endpoint
```
POST /api/chat
Body: { question: "...", patient_id: "..." }
Response includes trace_id for fetching detailed workflow
```

## Debugging with Workflow Traces

### Scenario 1: Slow Response
1. Check **Duration** in trace metadata
2. Look for slow **stages** (highlight longest bars)
3. Identify bottleneck agent or data source
4. Action: Check logs at `openmrs_chatbot/logs/` for details

### Scenario 2: Wrong Intent Detected
1. Check **Intent Classification** stage
2. Verify agent output matches your expectation
3. Review question keywords used
4. Action: May need to update keyword mappings in triage_agent.py

### Scenario 3: Incomplete Data Fetch
1. Check **Patient Data Fetch** stage
2. See how many records were retrieved
3. Look for errors in substeps
4. Action: Verify OpenMRS database connection

### Scenario 4: Unexpected Response
1. Check all **Processing** stage agents
2. Trace which agent generated the response
3. See if safety nets were triggered
4. Action: Cross-check agent logic in corresponding handler

## Architecture Overview

```
Frontend (http://localhost:5000)
    ↓
Flask API (app.py)
    ↓
ClinicalChatbot (main.py)
    ├── TriageAgent → Intent Classification
    ├── SQLAgent → Patient Data Fetching
    ├── MCPAgent → MCP Skills & APIs
    ├── KnowledgeAgent → Medical KB Search
    ├── ResponseAgent → Answer Formatting
    │
    ├── Specialized Handlers:
    │   ├── MedicationHandler (dosing, interactions)
    │   ├── AllergyHandler (contraindication checks)
    │   ├── ImmunizationHandler (vaccine records)
    │   ├── VitalsHandler (height, weight, BMI)
    │   └── MilestoneHandler (development tracking)
    │
    ├── WorkflowTracer ← Captures every step
    │
    └── Database (OpenMRS)
         ├── Patient records
         ├── Medications
         ├── Allergies
         └── Immunizations

Workflow Trace Data:
    Stored in memory (up to 50 queries)
    Available via /api/trace/{id}
    Displayed in frontend trace panel
```

## Common Questions & Answers

**Q: Why is the first query slow?**
A: Database connection pool needs warming up. Subsequent queries are faster.

**Q: Can I export workflow traces?**
A: Yes! The trace panel shows all data. Use browser DevTools → Network → api/trace to see JSON.

**Q: How long are traces kept?**
A: In memory for the last 50 queries. Server restart clears them. For persistence, logs are in chatbot.log.

**Q: Can I analyze traces programmatically?**
A: Yes! Fetch from `/api/traces` endpoint and parse the JSON. Duration and agent info are structured.

**Q: What if a query isn't traced?**
A: All queries go through tracing automatically. Check browser console for errors, or verify trace_id in response.

**Q: How do I understand the agent abbreviations?**
A: Hover over icons or check agent_involved field in trace JSON. Each agent is named descriptively.
