## WHAT HAS BEEN IMPLEMENTED

1. ✓ WORKFLOW TRACER MODULE (utils/workflow_tracer.py)
   - Captures every step from question input to response generation
   - Tracks intent classification, patient data fetching, agent processing
   - Measures duration of each stage and component
   - Stores last 50 queries in memory for analysis

2. ✓ TRACING INTEGRATION IN MAIN.PY
   - process_query() starts a trace and assigns it a unique ID
   - Each pipeline stage (doctor/patient) records its actions
   - Agents involved are logged with their inputs/outputs
   - Final response includes trace_id for frontend correlation

3. ✓ BACKEND API ENDPOINTS (app.py)
   - GET /api/trace/{trace_id} → Retrieve detailed workflow trace
   - GET /api/traces → List all recent traces
   - POST /api/traces/clear → Clear trace history (dev/testing)
   - POST /api/chat → Main chat endpoint (now includes trace_id)

4. ✓ FRONTEND WORKFLOW VISUALIZATION (chat.html + chat.js + style.css)
   - New "Backend Workflow Trace" panel below each response
   - Click panel header to expand/collapse detailed breakdown
   - Shows:
     * Trace ID, Duration, Intent Detected
     * Intent Classification stage (what was understood)
     * Patient Data Fetch stage (what was retrieved from database)
     * Processing stage (which agents were used, timing info)
   - Color-coded indicators for stage type and status

## KEY METRICS FROM TRACES

Each workflow trace captures:

Metadata:
  - Query ID
  - Question asked
  - User role (Doctor/Patient)
  - Patient ID
  - Timestamp
  - Total duration
  - Response length
  - Sources used

Per-Stage Info:
  - Stage name and description
  - Duration in milliseconds
  - Agents involved with their actions
  - Data sources accessed with record counts
  - Substeps showing execution flow

## WORKFLOW EXAMPLE: MEDICATION DOSAGE QUERY

Question: "What's the safe ibuprofen dose for my 10-year-old?"

Trace shows:
1. INTENT CLASSIFICATION (12ms)
   → Detects: MEDICATION_QUERY (with dose keywords)
   → Extracts: drug_name="ibuprofen"

2. PATIENT DATA FETCH (150ms)
   → OpenMRS: Patient weight (30kg), age (10 years)
   → Cache hit for patient info

3. PROCESSING (280ms)
   → MedicationController: weight-based calculation
      • 30kg × 10mg/kg = 300mg
      • Max single dose: 400mg → use 300-400mg
   → RxNormAPISkill: Validate with FDA
      • Confirmed safe for pediatric use
      • Frequency: every 6-8 hours
      • Max daily: 1200mg per day
   → ValidationAgent: Cross-reference allergies
      • No documented ibuprofen allergies
      • Safe to prescribe

Response: "300-400mg every 6-8 hours, max 1200mg/day"

## COMMON WORKFLOW PATTERNS YOU'LL SEE

Pattern A: DIRECT DATA QUERY (40-100ms)
  - Intent: VITALS_QUERY or specific data field
  - Fast path: Database only, no LLM
  - No agents needed
  - Example: "What is patient's age?"

Pattern B: SAFETY CHECK (200-350ms)
  - Intent: ALLERGY_QUERY
  - Fetches: Full allergy records + medication history
  - Validates: Cross-reactivity checking
  - Example: "Any drug allergies?"

Pattern C: CALCULATION QUERY (300-600ms)
  - Intent: MEDICATION_QUERY (with dose/dosage keywords)
  - Fetches: Patient demographics (weight, age)
  - Processes: Multiple agents (calculator, validator, API)
  - Example: "Correct paracetamol dose?"

Pattern D: KNOWLEDGE QUERY (2-5 seconds)
  - Intent: GENERAL_MEDICAL_QUERY
  - Sources: Medical knowledge base
  - AI: Ollama LLM generates detailed answer
  - Example: "What are side effects of...?"

## PERFORMANCE TIPS

From your workflow traces, you can optimize by noticing:

If something takes 500ms+ in Patient Data Fetch:
  → Database queries need optimization
  → Check for missing indexes in OpenMRS

If APIs take 1000ms+:
  → Network latency or API rate limits
  → Consider caching external API results

If response takes 5+ seconds:
  → Ollama LLM is a bottleneck
  → Consider using faster model or caching responses

## TROUBLESHOOTING

Q: Workflow trace panel doesn't appear?
A: Check browser console (F12) for errors. Verify trace_id in API response.

Q: Trace shows "no patient data" but query worked?
A: Direct data queries skip Patient Data Fetch. This is normal and expected.

Q: Duration seems inconsistent?
A: First query warms up database connection. Later queries are faster.

Q: Want to see detailed logs?
A: Check openmrs_chatbot/logs/chatbot.log for full execution trace.
